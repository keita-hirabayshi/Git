Gitの特徴 ・Gitはスナップショットを記録する(一部ではなく丸々保存する) → これにより、データの編集速度が上がる。特に大規模な開発の場合 ・コミットを辿る事で以前の状態に戻せる → gitはcommitする際に直前にcommitしたデータを記憶するので、最新のcommitから過去のcommitに遡れる Gitデータ移行の流れ データを編集し、スナップショットをローカルレポジトリに保存する → GitHub(リモートレポジトリ)へ移す Gitによるデータ管理（ローカル上）新規 Git addにてレポジトリにて圧縮ファイルを作成し、ステージにてファイル構成を追加する。 → Git commitにて先のデータ(ツリーA)とコミットAを保存する Gitによるデータ管理（ローカル上）追加 新たに追加されたファイルがステージに追加され、git commitにて新規にツリーBとコミットB()が追加 新規との違いは、コミットBに親コミットとしてコミットAの情報が保存されている点。これはGitの役割であるバージョン管理に直結する。 Gitによるデータ管理（ローカル上）変更 変更があったファイルのみ圧縮され、ステージの圧縮ファイルが上書きされる。Git commitにてツリーとコミットが保存される。 大きな違いは、データの変更の場合圧縮ファイルが上書きされる点。 Gitデータ構造のまとめ Gitではレポジトリに「圧縮ファイル」「ツリー」「コミット」ファイルを作成する事でデータを保存している。 → つまりデータを圧縮し、スナップショットで保存しているだけ ローカルの構造 ワークツリー(手元の作業) → ステージ(コミットする変更を準備) → レポジトリ(スナップショットを記録) 用語解説 コミット データをスナップショットで記録する行為 ワークツリー 自分の作業場 ローカルレポジトリ PC上でデータを保存する場（自分が変更する場合） 圧縮ファイル データを追加する場合、変更があったファイルのみ圧縮される ステージ gitでのデータ管理では、ワークツリーでのデータを一旦ステージに追加し、必要なものだけcommitしていくイメージ
